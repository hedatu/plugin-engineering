# 整体设计文档：Supabase + Cloudflare + Waffo + Chrome 插件会员系统

版本：2026-04-21

## 1. 总体架构

### 网站前端

网站托管在 Cloudflare Pages，承担：

- 官网首页 `/`
- 定价页 `/pricing`
- 登录页 `/login`
- 用户中心 `/account`
- 支付成功页 `/checkout/success`
- 支付取消页 `/checkout/cancel`
- 隐私政策 `/privacy`
- 服务条款 `/terms`

### Supabase

Supabase 负责：

- Auth
- Postgres
- Edge Functions
- RLS
- 会员授权
- 订单与支付数据

### Chrome 插件

Chrome 插件包含：

- popup
- options
- background / service worker
- content script

其中：

- background 统一持有 Supabase session 和敏感 token
- content script 不直接持有敏感 token
- 插件通过 Edge Functions 查询会员、扣减额度、创建 checkout

## 2. 产品目标

建立一个可复用的浏览器插件会员系统，支持：

- 一个网站 + 一个插件先跑通
- 后续多插件共享账号、支付、订阅、权益、额度体系
- 用户通过邮箱 OTP 登录
- 通过 Waffo Pancake 完成托管支付
- 根据 webhook 结果自动开通或撤销会员

## 3. 支付链路

```text
网站或插件点击升级
-> Supabase create-checkout-session
-> Waffo 托管结账页
-> 用户完成支付
-> Waffo webhook
-> Supabase 验签 / 去重 / 落库 / 更新 entitlement
-> 网站与插件刷新会员状态
```

## 4. 核心原则

- checkout session 只能由服务端创建
- metadata 必须包含 `userId`、`productKey`、`planKey`、`localOrderId`、`source`
- 如果接口支持，附带 `merchantProvidedBuyerIdentity`
- success 页面只轮询 `get-entitlement`
- success 页面不直接开会员
- 会员开通唯一依据是 Waffo webhook
- webhook 必须保留 raw body、验签、时间窗校验、去重和幂等处理
- Test / Production 的 product、plan、price、公钥、webhook URL 必须分离

## 5. 最小闭环

第一阶段需要跑通：

1. 用户登录
2. 创建 checkout session
3. 跳转 Waffo Test 支付
4. Waffo webhook 落库
5. 更新 `orders`、`payments`、`entitlements`、`processed_webhooks`
6. `/checkout/success` 轮询 `get-entitlement`
7. 插件和账户页显示最新权益
