## Assistant

可以做，但我建议你把这个技能定义成：**每天自动发现一个“有真实需求、已有用户量、但存在明显不满点”的细分扩展，然后产出一个差异化、单用途的新版本**。重点不是“抄一个插件”，而是“把现有插件当作需求证据”。Chrome Web Store 明确不允许重复体验、复制其他开发者工作、模板化的小变体；同时扩展必须是**单一目的**、能提供清晰价值。[Chrome for Developers+3Chrome for Developers+3Chrome for Developers+3](https://developer.chrome.com/docs/webstore/program-policies/spam-and-abuse)

先说最关键的现实约束。到 2026 年 4 月，Chrome Web Store 最新是 **V2 API**；它支持服务账号、分阶段发布、灰度百分比、取消待审核提交，但 **V2 不支持创建全新的 item，也不能通过 API 改可见性**。旧的 V1 仍然还能用到 **2026 年 10 月 15 日**，而且 V1 里还有 `insert` 新建 item 的能力，但它已经是过渡方案，不适合当你的长期核心能力。再加上一个开发者账号默认**最多只能公开 20 个扩展**，所以“每天都自动公开发一个全新插件”这件事，最好设计成“每天跑完整个发现和生成流程，但公开发布要过闸门”。[Chrome for Developers+3Chrome for Developers+3Chrome for Developers+3](https://developer.chrome.com/blog/cws-api-v2)

所以，我建议你的目标不是“日更公开上架”，而是：**日更选题、日更产物、择优发布**。这样既不会很快撞上 20 个扩展上限，也更不容易因为一堆相似条目被判成 repetitive content。[Chrome for Developers+2Chrome for Developers+2](https://developer.chrome.com/docs/webstore/publish)

你的整条流水线，最适合做成一个状态机：`DISCOVER → ENRICH → BRIEF → BUILD → QA → ASSETS → PUBLISH → MONITOR`。

第一步是 **找标的**。CodeX 先去抓 Chrome Web Store 的搜索页和类目页，建立候选池。筛选逻辑我会这样定：先看是否有一定用户量和评论量，再看评分是不是“有需求但还有明显提升空间”。经验上，太高分的产品往往已经很成熟，太低分的则可能是伪需求或质量太差；真正适合切入的，通常是“有安装量、有评论、近 90 天还有反复抱怨”的条目。Chrome 官方也明确说，商店的发现和排序会参考**评分、用户受欢迎程度，以及下载与卸载等使用数据**，所以你挑标的时不该只看安装量。[Chrome for Developers+1](https://developer.chrome.com/docs/webstore/discovery)

我会给候选项算一个 `Opportunity Score`，大致由四部分组成：需求强度、痛点密度、可差异化程度、实现可行性。你可以先用很简单的门槛起步：比如“至少 1 万用户、至少 100 条评论、评分 3.8 到 4.6、近 90 天负向评论占比不低、至少有 3 个可归类的反复抱怨”。这些阈值不是平台规则，是你内部的选题启发式。

第二步是 **挖用户意见**。这里不要只吃 Chrome Web Store 评论，因为 Google 明说了，商店里的评分和评论**不验证真实性**。所以你要把它们当“弱信号”，再叠加别的来源：扩展自己的 Support URL、公开支持站、GitHub issues、Reddit、独立站 FAQ、用户教程评论区。对你自己未来发布的扩展，建议把 Store 内建的 Support hub 打开，因为它会在工单里带上**扩展版本、浏览器类型、操作系统**，这对自动复现 bug 很有用。[Google 支援+1](https://support.google.com/chrome_webstore/answer/12225786?hl=en)

第三步是 **把“抱怨”变成产品 brief**。这里最重要的不是“我要做个更好的 XX”，而是只抓 **一个非常具体的 wedge**。例如，不做“万能标签页助手”，而做“帮招聘网站页面一键抽取候选人并导出标准化表格”；不做“万能翻译增强”，而做“只在某个 SaaS 后台把表格翻译成双语列”。Chrome 对单一目的要求很严，标题也要清楚、独特；summary 要短，且不要在摘要里直接碰瓷竞争对手。[Chrome for Developers+1](https://developer.chrome.com/docs/webstore/program-policies/quality-guidelines)

第四步是 **自动开发**。CodeX 在这一步不应该直接“复刻别人”，而应该基于 brief 重新实现。更稳的做法是：代码层面默认 greenfield；只有当对方本来就是公开仓库、许可证清晰时，才允许复用。功能层面坚持“最小可用 + 最少权限”。如果功能会碰到个人或敏感数据，就必须把数据用途绑定到单一目的上；Chrome 还要求，凡是处理个人或敏感数据的扩展，要在主页或隐私页展示 Limited Use 披露，对与核心功能不直接相关的数据处理还要在安装前做明确披露并取得同意。[Chrome for Developers+1](https://developer.chrome.com/docs/webstore/program-policies/disclosure-requirements)

第五步是 **素材和上架文案自动化**。这块完全可以交给 CodeX 配合图像模型去做，但要按商店规则出图：manifest 里最好至少有 **128x128** 图标，通常再补 **48x48** 和 **16x16**；商店 listing 至少要有 **1 张真实截图**，最好补满到 **5 张**；还要提供 **440×280** 的小 promotional image，若想争取更好展示位，再做 **1400×560** 的 marquee 图。官方也明确说，promo 图不要只是截图，应该主要表达品牌和核心价值。[Chrome for Developers+2Chrome for Developers+2](https://developer.chrome.com/docs/extensions/reference/manifest/icons)

第六步是 **QA 和合规闸门**。这一步不要省，因为很多扩展不是死在代码，而是死在审核。你的自动检查至少要覆盖：单一目的是否足够清晰、权限是否最小化、截图是否真实反映实际体验、标题和 summary 是否清楚且不堆关键词、隐私字段是否完整、是否需要测试账号和 test instructions。Chrome 对 listing 的要求很具体，summary 是用户在搜索结果里最先看到的那段，应该把最重要价值放进去；而截图必须展示真实体验。[Chrome for Developers](https://developer.chrome.com/docs/webstore/best-listing)

第七步是 **发布策略**。我的建议是把“创建新 item”单独做成一个半自动步骤，而把“上传新包、提交审核、灰度发布、查状态、取消提交、放大灰度”全部交给 V2 API。V2 现在支持服务账号、`STAGED_PUBLISH`、灰度百分比和取消 pending submission，所以你可以把发布做成很标准的流水线：先 private/beta，过烟测后再 staged publish，初始只放 5% 或 10%，稳定后再把 `deployPercentage` 拉高。如果是 Declarative Net Request 类扩展，部分改动甚至可以走 skip review。[Chrome for Developers+4Chrome for Developers+4Chrome for Developers+4](https://developer.chrome.com/blog/cws-api-v2)

第八步是 **监控和复盘**。Chrome Web Store 的 listing metrics 能看 installs、uninstalls、impressions，还能导出 CSV；另外还能接入 Google Analytics。但要注意，官方 GA 集成的数据保留只有 **两个月**，所以你最好每天自动把数据快照进你自己的库里。发布后的复盘不要只看下载量，还要看：卸载率、商店页转化率、差评主题是否改变、支持请求是否下降。Chrome 还明确提到，及时回复评论和支持请求有助于建立信任，评分和用户体验也会影响发现和排序。[Chrome for Developers+3Chrome for Developers+3Chrome for Developers+3](https://developer.chrome.com/docs/webstore/metrics)

如果你让我给这个技能定一个最实用的组织方式，我会拆成 7 个 agent：

1.  `Scout`：抓候选扩展，算机会分。

2.  `VoiceMiner`：抓评论和外部反馈，做聚类。

3.  `PM`：输出产品 brief，只允许 1 个核心 wedge。

4.  `Builder`：写代码、跑测试、出可安装包。

5.  `Compliance`：检查权限、隐私、single purpose、listing 文案。

6.  `Designer`：出 icon、截图、440×280 promo、可选 1400×560。

7.  `Publisher/Ops`：走 API 上传、灰度、状态跟踪、回滚和复盘。

每次运行，最终至少产出 6 份标准化成果物：`candidate_report.json`、`feedback_clusters.json`、`product_brief.md`、`qa_report.json`、`listing_assets/`、`publish_plan.json`。这样 CodeX 每天不是“想一想然后写点代码”，而是在执行一个可审计的流水线。

最后给你一个更接地气的落地顺序。第一版先别追求 100% 无人值守，先做 **80% 自动化**：自动找标的、自动抓反馈、自动生成 brief、自动开发、自动出素材、自动 QA；“创建新 item”先保留一个人工确认点。第二版再把已有 item 的更新发布全自动化。第三版如果你还坚持“每天新建新插件”，再评估要不要临时用 V1 的 `insert` 过渡，或者用浏览器自动化去操作 Dashboard——但长期看，最稳的还是把核心流程建立在 V2 上。[Chrome for Developers+2Chrome for Developers+2](https://developer.chrome.com/docs/webstore/api/v1?utm_source=chatgpt.com)

你真正要做的，不是“自动做插件”，而是**自动做一个合规的、以用户抱怨为输入的单用途插件工厂**。这样才更容易越做越准，而不是越做越像批量铺货。
